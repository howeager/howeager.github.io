01 — Introduction

Minecraft, my beloved, reigns supreme. It's so easy to just boot up a server, share your IP with your friends, and play for hours — until you can't. Or until one of your friends griefs the entire world and it suddenly turns into an unplanned anarchy server and everything crashes and burns. Amazing.

Over the years, me and my friends have drifted apart, moving into our own careers and lives. But the one thing that always manages to bring us back together? Minecraft.

One day, I got a message from a friend who lives halfway across the world — stationed in Qatar. He dropped a link in the group chat and said:

The Message "We should just buy a premade server. It's cheap." — followed by a Hostinger ad promising a fast, reliable, low-cost Minecraft server with almost no setup required.

I took a look. Honestly, from a consumer standpoint it made sense. Simple. Convenient. It "just worked." But that's exactly what bothered me.

I went into the field of information technology. I knew the ins and outs. I knew how to do this myself. And I took mild offense to the fact that he never asked me first. So instead of just saying no, I proposed a bet: if I could build a server setup that was on par — or better — than what Hostinger was advertising, he would have to pay me the subscription fee he was willing to spend. He agreed.

What followed was a deep dive into Proxmox, LXC containers, Tailscale, Velocity proxies, HashiCorp Vault, Prometheus, Grafana, Loki, and Gitea — all mapped against the DoD DevSecOps Tools & Activities framework. For a Minecraft server. You're welcome.

03 — Let's Be Cheap About It

The hardware powering all of this is an HP EliteDesk 705 G3 Mini — a small form factor business PC I grabbed off eBay for $70. Seventy dollars. Less than a year of Hostinger.

Hardware Specs HP EliteDesk 705 G3 Mini · AMD APU · 4 cores · 8GB DDR4 · 210GB storage · ~10–15W idle power draw

The RAM situation deserves its own paragraph. I started with a single 8GB DDR4 stick after discovering the EliteDesk 705 G3 is deeply opinionated about memory compatibility — a mismatched 16GB stick simply refused to boot. After some quality time in the BIOS, I ran solo on 8GB while a matching stick was inbound.

The "expensive" part of this whole project? My friends — after losing the bet — had to chip in and buy me a matching 8GB DDR4-2400 SO-DIMM. Retail price: about $15 on eBay. I cannot stress enough how much satisfaction came from that transaction.

Total hardware cost including RAM: roughly $85. The Hostinger plan they were about to buy costs more than that annually. We don't talk about ROI here. We talk about principles.

04 — DevSecOps Methodology

Before touching a single config file, I mapped this entire project to the DoD DevSecOps Fundamentals Guidebook: Tools & Activities (v2.0, March 2021). This is an unclassified, publicly available document defining eight phases of a DevSecOps software lifecycle and the security activities that must occur at each phase.

Most homelab writeups stop at "I spun up a server." This one maps every component to a phase of that framework — because that's how real enterprise DevSecOps works.

DoD Framework Reference Eight phases: Plan → Develop → Build → Test → Release & Deliver → Deploy → Operate → Monitor. Security is woven through every phase, not bolted on at the end.

Key principles carried into this build directly from the framework:

  • Zero Trust Architecture — perimeter security alone is not sufficient
  • SIEM/logging integration at every phase, not as an afterthought
  • Runtime defense and vulnerability management as baseline requirements
  • Infrastructure as Code for reproducible, auditable deployments
  • Secrets management — no hardcoded credentials, ever

05 — Plan Phase — Architecture Design

The DoD Plan phase covers system design, threat modeling, and configuration management planning. That meant deciding on network topology before writing a single line of config.

Core Design Decisions

Decision 01

LXC containers over VMs. Proxmox supports both full VMs and LXC containers. LXCs share the host kernel, making them significantly lighter and faster to clone. Each LXC is cloned from a base Debian 12 template — matching the DoD concept of "immutable infrastructure from master images."

Decision 02

Isolated internal network. All LXCs live on a private bridge (vmbr1, 10.10.10.0/24) completely separate from the home LAN. The Proxmox host acts as a NAT gateway. No LXC is directly reachable from the internet or local network.

Decision 03

Tailscale for zero-trust access. Rather than opening ports on the home router, Tailscale creates a WireGuard-based mesh VPN. NAT traversal means zero port forwarding required. ACLs enforce the privilege split: I get full access, friends get port 25565 only.

Threat Model

Even for a Minecraft server, threat modeling matters. Attack surfaces considered:

  • Unauthenticated access to Proxmox management interface — mitigated: Tailscale only
  • Credential exposure via hardcoded secrets — mitigated: HashiCorp Vault
  • Lateral movement between LXCs — mitigated: isolated bridge network
  • Privilege escalation via service accounts — mitigated: unprivileged LXCs, non-root users
  • Friend access to admin interfaces — mitigated: Tailscale ACL grants

07 — Build Phase — Configuration & Security Validation

The DoD Build phase requires SAST and dependency vulnerability checking as baseline requirements. For an infrastructure project, this translates to automated config validation and secret scanning on every commit.

Check 01 — TOML Syntax Validation

Validates velocity.toml for syntax errors before deployment. A malformed config file would silently break the proxy — this catches it at commit time.

Check 02 — Secret Scanning

Scans all config files for patterns matching hardcoded passwords, tokens, and keys. Maps directly to the DoD requirement for source code repository security plugins.

Check 03 — Default Credential Detection

Fails the pipeline if any file contains password=changeme or similar defaults. Simple but effective gate. 

name: Validate Configs
on: [push]
jobs:
  validate:
    runs-on: debian-latest
    steps:
      - uses: actions/checkout@v3
      - name: Validate TOML syntax
        run: python3 -c "import toml; toml.load('velocity/velocity.toml')"
      - name: Scan for hardcoded secrets
        run: |
          grep -r "password\s*=" . --include="*.toml" \
            --include="*.yml" --include="*.properties"
      - name: Check for default credentials
        run: |
          if grep -r "password=changeme\|password=admin" .; then
            exit 1
          fi

08 — Building the Core Cluster

With the design locked in, the actual build followed a repeatable pattern: clone base template, configure networking, install service, create non-root systemd user, enable and start.

Base LXC Template

A Debian 12 (Bookworm) LXC template was created with both Java 21 and Java 25 pre-installed via the Eclipse Temurin repository. Every subsequent LXC is a full clone of this template.

Network Isolation

A second bridge interface (vmbr1) was added to Proxmox with NAT masquerade rules. All LXCs attach to this bridge and route internet traffic through the host. The home LAN has no direct access to any LXC.

Service Users

Every service runs as a dedicated non-root system user. The systemd unit files use the User= directive to drop privileges before execution. 

[Service]
User=minecraft
WorkingDirectory=/opt/minecraft
ExecStart=/usr/bin/java -Xms1G -Xmx2G -jar server.jar nogui
Restart=always

Velocity Proxy Configuration

Velocity sits in front of both Minecraft nodes and handles player routing, load balancing, and the modern forwarding protocol. The forwarding secret shared between Velocity and each PaperMC backend ensures backends only accept connections from the trusted proxy.

Config Complexity Note The interdependencies between Velocity and PaperMC were the most frustrating part of this build. The player-info-forwarding-mode in Velocity must match the velocity.enabled setting in paper-global.yml, and both must reference the same forwarding secret. Getting this wrong silently drops players with no useful error. Lesson learned: read the source code, not just the docs.

09 — Deploy Phase — Running Minecraft in Containers

The DoD Deploy phase covers artifact deployment, infrastructure provisioning, and post-deployment security scanning. The deployment model here is immutable: to update a node, clone the template, configure it, and swap it in behind Velocity.

Two-Node Setup

Two identical PaperMC nodes (minecraft-1 at 10.10.10.20 and minecraft-2 at 10.10.10.21) run behind Velocity. Both are clones of the same base template with identical configurations. Velocity's try list handles failover automatically.

iptables Port Forwarding

Since LXCs are on an isolated internal network, Proxmox uses iptables PREROUTING rules to forward specific ports from the Tailscale interface to the appropriate LXC. 

# Minecraft — friends reach this via Tailscale
iptables -t nat -A PREROUTING -i tailscale0 -p tcp \
  --dport 25565 -j DNAT --to-destination 10.10.10.10:25565

# Admin services — you only
iptables -t nat -A PREROUTING -i tailscale0 -p tcp \
  --dport 3000 -j DNAT --to-destination 10.10.10.30:3000
iptables -t nat -A PREROUTING -i tailscale0 -p tcp \
  --dport 8200 -j DNAT --to-destination 10.10.10.40:8200

10 — Test Phase — Failure & Resilience Testing

The DoD Test phase includes acceptance testing, performance testing, and manual security testing. For this project, testing meant deliberately breaking things and confirming the system recovered.

Test 01 — Remote Access Verification

Confirmed Proxmox web UI was accessible over Tailscale on T-Mobile 5G with WiFi disabled. Zero port forwarding required on the home router.

Test 02 — Cross-Continental Connectivity

A friend stationed in Qatar connected to the server. No significant latency. Casual gameplay was smooth with no rubber-banding or timeout events. Tailscale's NAT traversal handled the distance without issue.

Test 03 — Access Control Enforcement

Confirmed friends could reach port 25565 but received connection refused on ports 8006, 3000, and 8200. Tailscale ACL grants working exactly as designed.

Test 04 — Service Recovery

Tested Restart=always behavior by forcibly killing the Minecraft process. Service auto-restarted within 10 seconds. Velocity's failover logic kept the session alive.

11 — Operate Phase — Backups & Reliability

The DoD Operate phase covers scaling, load balancing, and backup management. Proxmox handles this natively — LXC snapshots provide point-in-time recovery, and the template-based deployment model means any node can be rebuilt in minutes.

Sleep and suspend were permanently disabled on the host to ensure 24/7 availability:

systemctl mask sleep.target suspend.target \
  hibernate.target hybrid-sleep.target

CPU power management uses the powersave governor — the host idles at roughly 10–15 watts when no players are active, scaling up under load.

Every LXC runs both a Node Exporter (system metrics) and a Promtail agent (log forwarding). Velocity and Minecraft application logs are shipped to Loki with job labels. In Grafana, querying {job="velocity"} surfaces every player connection and disconnection in real time.

ISCM Mapping This stack maps directly to NIST SP 800-137's definition of Information Security Continuous Monitoring: maintaining ongoing awareness of information security, vulnerabilities, and threats to support organizational risk management decisions.

13 — Security Considerations (SecOps Layer)

Zero Trust Networking

The Tailscale ACL policy implements strict least-privilege. Two groups are defined: group:admins with full port access, and group:friends restricted to port 25565 only. The policy is version-controlled and any change is auditable. 

{
  "groups": {
    "group:admins": ["robasaldana@github"],
    "group:friends": ["friend@gmail.com"]
  },
  "grants": [
    {
      "src": ["group:admins"],
      "dst": ["proxmox"],
      "ip": ["tcp:22","tcp:8006","tcp:25565","tcp:3000","tcp:8200","tcp:9090"]
    },
    {
      "src": ["group:friends"],
      "dst": ["proxmox"],
      "ip": ["tcp:25565"]
    }
  ]
}

Secrets Management with HashiCorp Vault

The Velocity forwarding secret and RCON credentials are stored in Vault's KV-v2 secrets engine under the minecraft/ path. No credentials exist in any config file or repository. Vault is initialized with a single unseal key and auto-unseals on reboot via a systemd service.

Unprivileged Service Accounts

Every service — Minecraft, Velocity, Gitea, Vault, Prometheus, Grafana, Loki — runs as a dedicated non-root system user. If a service is exploited, the blast radius is contained.

Network Segmentation

The internal 10.10.10.0/24 network is completely invisible from the home LAN. LXCs can reach the internet through the NAT gateway but cannot be reached from the LAN without explicit iptables rules scoped to the Tailscale interface.

15 — Limitations & Reality Check

No honest writeup skips this section.

  • 8GB RAM is the ceiling. Four players is the comfortable operating range. The RAM upgrade (courtesy of my friends) brings that to 16GB.
  • Single point of failure. The HP EliteDesk is the only physical machine. If it goes down, everything goes down. This is a homelab, not a data center.
  • No automated backups yet. World data lives on the LXC filesystem. Proxmox snapshots exist but aren't scheduled.
  • Vault unsealing. After a reboot, the auto-unseal script handles the key — but storing an unseal key on disk is a deliberate homelab trade-off, not a production recommendation.

17 — Conclusion

A $70 eBay machine. A bet. A friend in Qatar. And somehow, a production-grade DevSecOps homelab mapped to a DoD framework.

The real takeaway isn't the Minecraft server — it's that every component of this build has a direct analog in enterprise infrastructure. Proxmox is VMware. Tailscale ACLs are network access policies. Gitea Actions is Jenkins. Vault is CyberArk. Prometheus and Grafana are Datadog. Loki is Splunk.

You don't need a cloud budget or enterprise hardware to practice real DevSecOps. You need a clear methodology, the willingness to read documentation, and ideally a friend who almost bought a Hostinger server.
